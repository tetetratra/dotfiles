## 目的

タスクの現状を分析し、次に実行すべき作業を提案します。

一般に、あるタスクを与えられたとき、それを遂行するための抽象的な手順を考えるところから始めると思います。
ここでいう手順というのは、具体的な作業ではなくメタ的な作業単位のことであり、
「このタスクは目的が十分に明確化されていないから、まずは『背景・目的の整理』からはじめよう」
「このタスクはやるだけのタスクだから、いきなり『実装』からはじめよう」
などの『背景・目的の整理』『実装』を指しています。(実際にどう整理や実装をやっていくかを指しているわけではありません)

まずは抽象的な手順を考え、その後に具体的な作業内容を考え、それをユーザーに提案します。

## 手法

- 用語の整理
  - ステップ：「要件定義」「実装」などの、タスクを進めるための抽象的な作業単位
  - ロードマップ：「要件定義 → 設計 → 実装計画 → 実装」といった、タスクを進めるためのステップの順序付け


### Step 1: ロードマップの構築

ロードマップを構築します。
`## フローチャート` セクションに従って、タスクのロードマップを構築してください。

### Step 2: ロードマップの確定

[注意] Step 1が完了してから着手してください

決定した各ステップごとに、実施にあたって使うスキルを検討してください（スキルを使わない判断をしてもよいです）。
特に `dev-step--` というprefixが付いているスキルを対象に、そのステップを実施するために必要なスキルがないかを中心に検討してください。

その後、以下の出力フォーマットで提案内容を出力してください。
そして、この内容で問題ないかを、あなた（AI）の自律レベル次第でユーザーに確認してください。

```
## フローチャート結果
- 手順1: {回答した判定フローチャートの質問} → {yes/no などの判断}
  - 判断理由: {判断理由を簡潔に}
  - 追加したステップ: {追加したステップ名があれば記載}
- 手順2: {回答した判定フローチャートの質問} → {yes/no などの判断}
  - 判断理由: {判断理由を簡潔に}
  - 追加したステップ: {追加したステップ名があれば記載}
...

## ロードマップの提案
1. {ステップ名} ({使用予定のスキル}): {そのステップで行うこと}
2. {ステップ名}:({使用予定のスキル}) {そのステップで行うこと}
...

{ロードマップの提案理由について、まとめのコメントがあればここに記載}
```

## フローチャート

- 手順1: 与えられた依頼やチケット中に、タスクの背景や目的が書いてありますか？以下のチェックリストで確認してください
  - チェックリスト
    - タスクのビジネス的な意義・必要性が書いてある
    - タスクが起票された経緯が書いてある
    - タスクで達成したいことと、その理由が書いてある
  - すべてチェックできる: 次へ
  - 1つでもチェックできない: 「背景・目的の整理」を追加し、終了

- 手順2: 今回のタスクを進めるために、あなた（AI）は以下の情報を持っていますか？以下のチェックリストで確認してください
  - チェックリスト
    - 関連するシステムの動作フローを理解している
    - データ構造やテーブル構成を把握している
    - 使用する技術スタックの特性を理解している
    - 類似の実装例や前例を知っている
  - すべてチェックできる: 次へ
  - 1つでもチェックできない: 「既存システム・関連技術の調査」を追加し、終了

- 手順3: 必要であればユーザーに、今回のタスクの関連システムと関連技術を伝え、理解しているか質問して下さい。
  - 質問不要と判断した場合: 次へ
  - 理解していた場合: 次へ
  - 理解していなかった場合: 「関連システムの理解促進」や「関連技術の理解促進」を追加し、終了

- 手順4: 「要求定義」「要件定義」「対応方針決め」のうち、未実施かつ非自明なものはありますか？
  - ある：それらをその順序で追加し、次へ
  - ない：次へ

- 手順5: 手順4で追加したステップのうち、下位（あとに追加したもの）が上位（先に追加したもの）に強く依存している・もしくは強く依存しそうなものはありますか？
  - 下位が上位に依存している例
    - 仕様（要求定義・要件定義）を決めるときに、そもそも現在の技術でどこまでできるのか分からなくて仕様を決められなくなりそう
    - 目的を決めるとき、どれくらい大変かによっては目的の再定義が必要になりそう
  - ある：その上位ステップよりも前の箇所に、その依存が解消されるような任意のステップを追加してください
  - ない（or Q4でなにも追加していない）：次へ

- 手順6: タスクを完遂させるまでに残り必要なステップを考え、追加してください
  - 例: 「設計」「実装計画」「実装」「動作確認」「PR作成」など

## ステップ分けの判断基準

ステップ分ける際は、以下の区切り方を参考にしてください。

- 意思決定単位で区切る（例: 「技術選定」→「設計」→「実装計画」）
- 依存関係の前後で区切る（例: 画面仕様の合意が前提なら「UI設計」→「実装」）
- 成功基準が変わる地点で区切る（例: 動作検証が目的なら「実装」→「検証環境で動作確認」）
- リスクが変わる地点で区切る（例: 新規ライブラリ採用なら「技術検証（PoC）」→「実装」）
- 合意/レビューが必要な地点で区切る（例: 設計レビューが必須なら「設計」→「設計レビュー」→「実装」）

## チェックポイント
以下に気をつけてください。
もし間違えていることに気付いた場合、やり直してください。

- フローチャートに記載の各質問に対して、「終了」に到達した場合を除いて、途中でスキップなどしていないこと
- 逆に、「終了」に到達している場合は、そこでフローチャートを終了していること
