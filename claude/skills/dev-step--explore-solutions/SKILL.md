タスクに対する解決方針を体系的に検討し、実装アプローチを決定します。

**注意**
このコマンドは技術的な詳細設計を行うものではありません。
高レベルな解決アプローチと大まかな作業フローを整理することが目的です。
詳細な技術設計は以降のステップで行います。

## 目的

- タスクの解決アプローチを複数案で比較し、推奨方針を短く決める
- 高レベルな対応方針と作業フローを共有し、詳細設計への接続を明確にする

## 手順

1. **方針案の提示**:
   - 検討すべき項目が複数ある場合(例: 認証方式、DB設計、UI実装など)、それぞれの項目について案を提示する
   - ベストな案が明確な場合は、その1案を詳細に提示し、他の選択肢は簡潔に補足する
   - 複数案を比較検討すべき場合は、実現可能なアプローチを2〜3案挙げ、方向性レベルで説明する
2. **メリデメ整理**: 案ごとに差別化できるメリット・デメリットを整理し、トレードオフを示す。各項目は「なぜそうなるのか」の根拠を含めて書く（単語の羅列にしない）。書式は比較の性質に応じて使い分ける:
   - **表形式**: 定量的に比較できるもの（料金、性能値など）、浅い比較で済むもの、比較対象が3つ以上あるもの。一番右の列に「備考」列を設ける（空でもよい）
   - **箇条書き**: 定性的で深い考察が必要なもの、案ごとの文脈や背景の説明が重要なもの
3. **推奨案決定**: 実装コストとレビュー工数、リスクの小ささを加味して推奨案を明示し、推奨理由を簡潔に述べる（条件付き可）
4. **書式整形**: `## 対応方針` 配下に（観点が複数あれば）観点ごとに案を配置する
5. **自己チェック**: 下記チェックリストで冗長さと妥当性、リンクの有無、粒度を確認する

ユーザーからフィードバックがあり、新しい方針案を設ける場合は、(既出の方針案を上書きするのではなく) 新たな案として追加してください。

## 成果物例

以下のフォーマットで出力してください（内容はあくまで例示です）

```
## 対応方針

### 観点A：認証方式について
- 案1：OAuth 2.0 + JWT (推奨)
  - 方針：標準プロトコルを採用し、トークンベース認証で実装。既存セッション認証と並行稼働させる
  - メリット
    - トークンベースのためネイティブアプリからも同じ認証フローを使え、モバイル対応が容易
    - セッションをサーバー側で持たないため、サーバー台数を増やしてもセッション共有の考慮が不要
    - OAuth 2.0は広く普及しており、ライブラリやドキュメントが豊富で実装コストが低い
  - 推奨理由：将来性とモバイル対応を優先。標準プロトコルでレビュー工数も抑えられる

- 案2：SAML
  - 方針：エンタープライズ向け基盤を導入し、SSOで統合
  - メリット
    - エンタープライズ環境では事実上の標準であり、既存の社内IdPとの互換性が高い
  - デメリット
    - XMLベースでペイロードが大きく、モバイルアプリでの実装が複雑になりやすい
    - リダイレクトフローに依存するため、SPAやネイティブアプリとの統合に技術制約が多い

### 観点B：データベース設計について
- 案：既存Userテーブル拡張 + 認証情報を別テーブル化 (推奨)
  - 方針：`users`テーブルに最小限のカラム追加、認証トークンは`oauth_tokens`テーブルで管理
  - 推奨理由：既存データとの整合性を保ちつつ、認証情報を分離。マイグレーションリスクが低い

- 検討したが採用しなかった選択肢
  - 完全新規テーブル：既存データ移行が大規模になり、リスクが高い
  - NoSQL移行：既存RDBMSとの整合性が困難、学習コストが高い

### 観点C：ロールアウト戦略について
- 案1：Flipperによる段階的ロールアウト (推奨)
  - 方針：`Feature::NEW_AUTHENTICATION`で制御し、特定ユーザー→部署→全体の順に展開
  - メリット
    - 既にFlipper導入済みのため追加の実装コストが最小限で済む
    - フラグをオフにするだけで即座にロールバックできるため、障害時の復旧が速い
  - デメリット
    - 新旧2つの認証方式が並行稼働する期間が長くなり、両方のコードパスの保守が必要

- 案2：Blue-Green Deployment
  - 方針：本番環境を複製し、新環境で全面切り替え後にDNS切り替え
  - メリット
    - DNS切り替えで一括移行するため、新旧並行稼働の期間がなくコードパスがシンプル
  - デメリット
    - 環境を2面維持するためインフラコストが2倍になる
    - 切り替え中のデータ同期が複雑で、不整合のリスクがある

- 検討したが採用しなかった選択肢
  - カナリアリリース：Kubernetesインフラが未整備のため採用困難

### 観点D：SaaS候補の比較（表形式の例）

定量的な比較や候補が複数ある場合は、表形式でまとめる。

| 項目 | サービスA | サービスB | サービスC | 備考 |
|---|---|---|---|---|
| 月額料金（目安） | $500/月 | $300/月 | $800/月 | いずれも50TBプラン |
| 無料枠 | あり（10GB） | なし | あり（5GB） | |
| SLA | 99.99% | 99.9% | 99.99% | |
| 日本リージョン | あり | なし | あり | レイテンシに影響 |
| **推奨** | **◎** | | | 料金・SLA・リージョンのバランスが最も良い |
```

## チェックポイント
- [ ] 複数の検討項目がある場合、各項目について案を提示しているか
- [ ] ベスト案が明確な場合、無理に複数案を並列せず、補足的に他の選択肢を示しているか
- [ ] 案ごとの差分とトレードオフが明確か
- [ ] 技術詳細に踏み込みすぎず、方向性レベルに留まっているか
- [ ] 推奨案と推奨理由が簡潔に示されているか
- [ ] 参考資料や技術を初出でリンクしているか
- [ ] `## 対応方針` としてそのまま貼れる体裁か
