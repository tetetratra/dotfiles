タスクを進めるうえでの前提条件・制約・達成条件を明確化します。

**注意**
このコマンドは、タスクの背景情報を受けて、実現に必要な条件を整理するものです。
解決策の検討は行わず、「何が必要か」「何を守るべきか」「何を達成すべきか」を明確にすることが目的です。

**重要: 開発一般に当てはまることは記載しないでください。このタスクに固有の条件のみを記載してください。**

## 出力物

このコマンドでは、以下の3つのカテゴリで出力を行います：

- **返答**: 作業完了の報告や調査結果の要約など、フロー型の出力（会話として返す）
- **成果物**: ドキュメントや整理結果など、ストック型の出力（後から参照できる形で返す）
  - 成果物の保存先について指示がある場合は、指定された箇所に書き込む
  - 指示がない場合は、通常のレスポンスとして返答の後に続けて出力する
- **作業**: システムへの変更や実行など、副作用を伴う作業

本コマンドの出力：

**返答**: 整理完了を報告する

**成果物**: `## 前提条件・制約・達成条件` セクションの本文をレスポンスとして出力する

**作業**: なし

## 目的

- このタスク特有の前提条件を洗い出す（開発一般の前提は除く）
- このタスク固有の制約を明確化する（プロジェクト標準は除く）
- 目的に含まれていない、このタスク特有の達成条件を定義する（該当がなければ省略）

## 入力で確認すること

- 対象タスクのリンクと概要
- タスクの背景情報の内容（特に目的セクションを確認し、達成条件との重複を避ける）
- 技術的制約（使用技術/フレームワーク/インフラ/互換性）
- ビジネス制約（予算/人員/期限/承認プロセス）
- 運用制約（無停止要件/ロールバック方針/監視要件）
- セキュリティ・コンプライアンス要件
- 関連システム・サービスへの影響範囲

上記について特に記載が無ければ気にしなくて良いです。

## 手順

1. **前提条件の洗い出し**: このタスク特有の前提条件のみを列挙（開発一般の前提は除く）
   - このタスクでのみ必要な特定のリソース
   - 先行して完了すべき特定のタスクや承認事項
   - 通常とは異なる権限やアクセスが必要な場合のみ
2. **制約の整理**: このタスクに固有の制約のみを列挙（プロジェクト標準は除く）
   - このタスクでのみ適用される技術制約（互換性維持/特定の技術禁止など）
   - 通常と異なる期限や予算、特別な承認フロー
   - このタスク特有の運用制約（特定時間帯のみデプロイ可能など）
3. **達成条件の確認**: 定義済みの目的を確認し、目的に含まれていない達成基準がある場合のみ記載
   - 目的で既に述べられている内容は記載しない
   - 特別な検証手順や承認プロセスなど、目的に含まれていない条件のみ
   - 該当がなければ達成条件セクション自体を省略
4. **書式整形**: 各セクションを箇条書きで整理し、該当がないセクションは省略
5. **自己チェック**: チェックポイントで「開発一般ではなくタスク固有か」「目的と重複していないか」を確認する

## 成果物例

```
## 前提条件・制約・達成条件

### 前提条件
- [タスク#123](https://example.com/123) のマイグレーションが完了している
- 既に外部決済API(XXX SaaS)のSTGアカウントでテスト可能な状態になっている

### 制約
- 期日はN月N日までに完了する必要がある（現在M月M日から残りL日）
- どのロールに対して権限を付与するかを決めておく必要がある
```

## チェックポイント

- [ ] 開発一般ではなく、このタスクに固有の条件のみが記載されているか
- [ ] 達成条件は既に定義済みの目的と重複していないか
- [ ] プロジェクト標準や通常の開発プロセスに含まれる自明な制約を除外しているか
- [ ] 各項目に「なぜこのタスク特有なのか」の理由が明確か
- [ ] 前提条件は開始前に確認・準備可能な項目になっているか
- [ ] 制約は実行中に遵守すべき具体的な条件として記載されているか
- [ ] 該当する条件がないセクションは省略されているか
- [ ] 不明点や仮定が明示されているか
- [ ] ですます調で、箇条書き末尾に句点がないか
- [ ] `## 前提条件・制約・達成条件` としてそのまま貼れる体裁か
