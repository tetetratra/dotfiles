実装した機能に対して、どのような動作確認が必要かを体系的に洗い出し、確認項目リストを作成します。

## 目的

- 実装内容を踏まえて必要な確認項目を整理する
- チェックボックス形式で3〜5項目程度にまとめ、後続の検証実行や PR 作成で活用できるようにする

## 入力で確認すること

- 対象タスクのリンクと概要
- 実装内容（何を実装したか、どのファイルを変更したか）
- 変更の影響範囲（既存機能への影響があるか）
- タスクの要件や達成条件
- 検証環境（開発環境、QA環境）
- 情報が不足している場合は、着手前に質問を箇条書きする

## 出力の要件

- レスポンスとして直接返す（ドキュメントファイルは作成しない）
- チェックボックス形式（`- [ ]`）で主要な確認項目を3〜5項目程度列挙
- 各確認項目は具体的で実行可能な内容にする
- 細かすぎる項目は避け、大きな確認観点に絞る

## 手順

1. **実装内容の確認**: 何を実装したか、どの機能を変更したかを理解する
2. **確認観点の整理**: 新機能の動作、既存機能への影響など、このタスクで重要な確認観点を洗い出す（基本的には正常系を試す、条件の網羅はしない）
3. **環境の考慮**: 以下の基準で確認環境を決定する
   - 基本的には開発環境とQA環境の両方で確認
   - QA環境で大変なこと（大量データの準備、複雑なセットアップ等）は開発環境のみで確認
   - 開発環境で詳細にテストしている場合、QA環境では最低限の確認でよい
4. **確認項目の作成**: 3〜5項目程度（RPの性質に応じて増減してよい）にまとめ、チェックボックス形式で出力する。必要に応じて環境を明記する

## 出力例

```
## 動作確認項目

開発環境で、以下を確認する
- [ ] 管理画面での日時入力が正しく動作することを確認
- [ ] 既存データの表示が正しく行われることを確認
- [ ] 大量データでのパフォーマンスに影響がないことを確認

QA環境で、以下を確認する
- [ ] 管理画面での日時入力が正しく動作することを確認
- [ ] 既存データの表示が正しく行われることを確認
```

## チェックポイント

- [ ] 実装内容を正しく理解しているか
- [ ] 主要な確認項目が3〜5項目程度にまとめられているか
- [ ] 各確認項目が具体的で実行可能な内容か
- [ ] チェックボックス形式（`- [ ]`）で記述されているか
- [ ] 正常系を中心とした確認項目になっているか
