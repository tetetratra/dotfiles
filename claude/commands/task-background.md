タスクのビジネス的背景・目的を体系的に定義し、チケットに記載します。

## 目的

- タスクの背景とビジネス的意義を短く言語化し、誰でも「なぜ今やるのか」を理解できるようにする
- ビジネス価値（根本価値/中間価値）と直接的な開発作業を論理的につなぐ

## 入力で確認すること

- 対象タスクのリンクと概要
- ユーザー課題・事業課題・発生している問題や機会
- なぜ今対応するのか（優先度/リスク/デッドライン）
- 期待する効果（ユーザー価値/業務効率/収益・リスク低減など）
- 関連ドキュメント・議論・過去対応・ステークホルダー/制約
- 情報が不足している場合は、先に箇条書きで質問するか、明示的に仮置きを記載する

## 出力の要件

- `## 目的・背景` に貼り付ける本文のみを出力
- ですます調で3〜5文、文ごとに改行し、段落間に空行を入れる
- 見出し/太字は使わず平文。必要ならバッククォートで囲む
- 初出の資料/チケットにのみリンクを付与
- 根本価値 → 中間価値 → 直接的作業の流れを必ず文章化する

## 手順

1. **タスクの把握**: タスク内容から目的物・変更箇所を一文で整理する
2. **ビジネス課題の特定**: 「なぜ必要か」「放置すると何が起きるか」を掘る
3. **背景情報の収集**: 関連チケット、議論、仕様、過去事例、制約、ステークホルダーを確認し、根拠となる情報源を把握
4. **価値連鎖の整理**:
   - 根本価値: ユーザー/事業に対する最終的価値
   - 中間価値: プロセスや機能改善による効果
   - 直接的作業: 今回の開発・対応内容
5. **本文作成**: 現状課題→必要性→期待効果→本タスクでの対応の順で、上記価値連鎖を含めて簡潔に書く
6. **配置**: 指定のドキュメント/チケットに `## 目的・背景` を追加し本文を配置する
7. **自己チェック**: 下記チェックリストで内容と形式を確認する

## 例

```
## 背景・目的

メンテナンス時のアラート表示が[hogehoge](https://example.com/hoge)経由に依存し、都度開発チームのサポートが必要なため運用負荷が高く、Bizチームが自立できていません。

本対応でアラートをコード側で処理する仕組みに切り替え、Bizチームが手元で即時に設定変更できるようにし、運用リードタイムを短縮します。

これにより障害時の顧客影響を素早く抑制し、事業の信頼性と顧客満足を高めます。
```

## チェックポイント

- [ ] ビジネス上の「なぜ」と優先理由が明確か
- [ ] 根本価値→中間価値→直接的作業の連鎖が示されているか
- [ ] ユーザー/事業への具体的な効果が述べられているか
- [ ] 3〜5文、ですます調で、文ごと改行・段落に空行があるか
- [ ] 関連資料やチケットを初出でリンクしているか
- [ ] `## 目的・背景` セクションとしてそのまま貼れる体裁か
