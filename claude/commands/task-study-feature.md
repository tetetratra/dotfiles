$ARGUMENTS

## 目的

ユーザーにとって未知のシステムや機能について、理解しやすい文章にまとめ、ユーザーの理解を促します。

## 手順

以下のテンプレートに従って、指定の機能についてまとめてください。
不明点があれば適宜調べてください。それでも分からない箇所があった場合、憶測で記述するのではなく、不明であることを明記してください。

## テンプレート

以下はメタ的な指示です。

`{}` メタ変数です。
`//` コメントです。具体例や、中身の指示などが書かれています。
`...` 繰り返してほしい箇所です。この記号が書かれていない場合であっても、必要に応じて繰り返してください

```
# {システム名 / 機能名} 概要理解ドキュメント // 例：リアルタイムQ&A・モデレーション機能

## 1. ビジネスコンテキスト
// 意図：システムの存在意義（Why）を明確にし、解決すべき課題と提供価値の目線を関係者で合わせる

// 例：
// 解決する課題: 数千人規模のイベントで、不適切な質問を排除しつつ円滑に質疑応答を行いたい
// ターゲットユーザー: 講演会の視聴者、運営モデレーター、登壇者
// 価値定義: 双方向性の高い講演会を、低リスクかつ効率的に運営できる

* **{観点}**: {説明}
* ...

## 2. 論理構造・機能俯瞰
// 意図：システムを大きな「役割の塊」として捉え、各要素の責務と境界線を可視化する

**主要なモデル・コンポーネント**
// システムの概要を理解する上で最低限押さえておくべきモデル・コンポーネントを列挙し、その役割・責務を説明する

- **{名前}** // 例：投稿UI 
  - {その役割・責務} // 例：視聴者からの入力を受け付け、バリデーションを行う
  - {追加の補足情報があれば} // 例：匿名投稿の有無や文字数制限などのルールを適用する
  - ...
...

## 3. 主要なデータフロー
// 意図：静的な構造図に「時間軸」を加え、実際のユーザー操作に伴うデータの変遷や連携の順序を明らかにする

* シナリオ: {対象とするユースケース} // 例：視聴者が投稿した質問が承認され、全体に公開されるまで
    1. {ステップ1}: {処理や状態の変化} // 例：視聴者の投稿アクションにより、バリデーションを経て「未承認」状態でDBへ保存
    2. {ステップ2}: {処理や状態の変化} // 例：バックエンドからモデレーター管理画面へ、新着質問の通知がリアルタイムに送信される
    3. {ステップ3}: {処理や状態の変化} // 例：モデレーターの承認操作により、DBのステータスが更新され、全視聴者へ配信イベントが発行される
    4. {ステップn}: {処理や状態の変化} // 例：各クライアントのUI上で、対象の質問がリストに追加表示される

{シーケンス図}
// 図はアスキーアートではなくmermaidで書いてください

* シナリオ: ...

## 4. 詳細設計
// 意図：設計思想をコードやDB構造の具体的な構成案に落とし込み、開発者が迷いなく実装・修正に着手できるようにする

### クラス設計
// ファイルのパスごとに1段階層ずつインデントを下げてください

- {directory_name}/
  - {directory_name}/
    - {file_name}: {内容と役割を説明} // 例：questions_controller.rb: 視聴者からの投稿およびモデレーターの承認アクションを制御する
      - #{method_name}: {重要と判断したメソッドの簡潔な説明} // 例：#approve: 質問のステータスを公開に変更し、配信イベントを発火させる

### テーブル設計 // テーブルが関係ない機能の場合は省略すること

{ER図を記載}
// ER図はアスキーアートではなくmermaidで書いてください
// ER図には、以下の情報を記載すること
// - テーブルについて、そのテーブルの概要を説明すること
// - カラムについて、そのモデルによって表現しているドメインに関する重要なカラムには、カラムの説明コメントを記載すること

### API設計 // APIが関係ない機能の場合は省略すること

- {METHOD} {/api/endpoint}: {用途と処理内容を説明} // 例：POST /api/v1/questions: 視聴者が新しい質問を投稿する
  - リクエスト: { {request_body_sample} } // 例：{ "content": "質問内容...", "is_anonymous": true }
  - レスポンス: { {response_body_sample} } // 例：{ "id": "q_123", "status": "pending" }

### {任意の見出し} // その他必要に応じて、あると良い見出しを追加してください
```
