タスクに対する解決方針を体系的に検討し、実装アプローチを決定します。

**注意**
このコマンドは技術的な詳細設計を行うものではありません。
高レベルな解決アプローチと大まかな作業フローを整理することが目的です。
詳細な技術設計は `task-design.md` で行います。

## 使用例

```
/task-solution Notionのこのチケット https://notion.so/xxxxx に対して対応方針セクションを作成してください
```

## 手順

1. **目的特有の観点を整理する**
   - その課題/機能に特有の制約・要件を洗い出す
   - 一般論（「保守性の高いコードを書く」等）ではなく、この具体的な目的での重要観点を明確化
   - ビジネス要件、技術的制約、パフォーマンス要件、運用要件等から導出

2. **解決案を提示する**
   - 実現可能なアプローチを挙げる（1案でも複数案でも可）
   - 技術的詳細には踏み込まず、方向性・アーキテクチャレベルで記述

3. **各案のメリット・デメリットを整理する**
   - 案同士を差別化できる要素を中心に記載
   - トレードオフを明確にする（パフォーマンス vs 保守性、開発速度 vs 拡張性等）
   - 最低限重要な差分のみに絞る

4. **推奨案を決定する**
   - 推奨案を選定する
   - 選定理由を簡潔に説明
   - 状況によっては「どちらでも良い」という判断も可

5. **対応方針セクションの作成**
   - 指示された方法でタスクの目的・背景セクションを実際に記載する
   - 出力例を参考に、`## 対応方針` セクションを記載する
   - 既存の構造を維持しながら新しいセクションを挿入


## 出力例

```
## 対応方針

### 観点
- セキュリティレベル（金融系なので高レベル要求）
- レガシーシステムとの連携必須
- モバイルアプリからの利用頻度が高い
- 既存ユーザーの移行を無停止で行う必要

### 方針案

**案1: OAuth 2.0 + JWTベース認証**
- 方針
  - 新規認証基盤を構築
  - レガシーシステムはAPIゲートウェイ経由で連携
  - 段階的な移行を実施
- メリット
  - モバイルアプリに対応しやすい
  - モダンな実装ができる
  - スケーラビリティが高い
- デメリット
  - レガシーシステム連携に改修が必要
  - 学習コストがかかる

**案2: SAMLベース認証**
- 方針
    - エンタープライズ向け認証基盤を導入
    - 既存システムの認証機能を置き換え
    - シングルサインオンで統合
  - メリット
    - エンタープライズ標準として確立されている
    - 既存システムとの対応が良好
    - 移行リスクが小さい
  - デメリット
    - モバイル実装が複雑になる
    - 技術的制約が多い

### 推奨案

**案1 (OAuth + JWT)** を推奨

理由: 将来性とモバイル対応を優先。
レガシー連携の改修コストはあるが、長期的なメンテナンス性と拡張性で優位。
金融系要件もJWTの適切な実装で対応可能。

### 方針

(最終的な方針はユーザーが判断し、ここに記載する)
```
