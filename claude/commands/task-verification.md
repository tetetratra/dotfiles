$ARGUMENTS

実装した機能に対して、動作確認の計画作成と検証実行を行い、結果を記録します。

## 出力物

計画と実行の完了報告、動作確認項目のチェックリスト（計画時/実行時）を返す。

## 目的

- 実装内容を踏まえて必要な確認項目を整理する
- 確認項目を1つずつ実施し、結果を記録する
- 確認結果を記録し、PR 作成時に活用できるようにする

## 入力で確認すること

- 対象タスクのリンクと概要
- 実装内容（何を実装したか、どのファイルを変更したか）
- 変更の影響範囲（既存機能への影響があるか）
- タスクの要件や達成条件
- 検証環境（開発環境、QA環境）
- 検証計画で作成した確認項目リスト
- テストデータの準備状況
- 情報が不足している場合は、着手前に質問を箇条書きする

## 検証実行における重要な指示

### 検証対象環境について
- **検証計画では開発環境とQA環境の確認項目を整理してよい**
- **実際の検証実行は開発環境でのみ行う**
- QA環境での検証は実施しない（人間が別途行うため）
- 検証計画にQA環境での確認項目が含まれていても、実行は開発環境のみで行う

### 環境の準備と操作
- ローカルでRailsサーバーを起動する（`rails server` など）
- フロントエンドのビルドやサーバーを起動する（`npm run dev` など）
- 必要なサービス（データベース、Redis等）が起動していることを確認する
- サーバー立ち上げやビルドの手順がわからない場合は、リポジトリの CLAUDE.md や README.md 等のドキュメントを参照する

### ブラウザ操作による検証
- Playwrightを使用してブラウザを起動し、実際のユーザー操作をシミュレートする
- 画面の表示、ボタンのクリック、フォーム入力などを実際に操作して確認する
- 必要に応じてスクリーンショットを取得する

### 検証完了後の後片付け
- すべての確認項目の検証が完了したら、起動したサーバーやプロセスを停止する
- 確認項目ごとに起動・停止を繰り返すのではなく、一連の検証が終わってから停止する

## 手順

1. 実装内容を把握し、確認観点を整理する
2. 検証計画として3〜5項目のチェックリストを作成する（必要なら環境を明記）
3. 開発環境で検証を実施し、結果を記録する
4. 問題があれば修正し、再確認して結果をまとめる
