$ARGUMENTS

タスクの解決方針を検討する際に、どのような観点で分析すべきかを明確化します。

**注意**
このコマンドは一般的な観点（セキュリティ、コスト、パフォーマンスなど）を網羅的に列挙するものではありません。
このタスク固有の特別な観点・価値観を抽出することが目的です。

## 出力物

このコマンドでは、以下の3つのカテゴリで出力を行います：

- **返答**: 作業完了の報告や調査結果の要約など、フロー型の出力（会話として返す）
- **成果物**: ドキュメントや整理結果など、ストック型の出力（後から参照できる形で返す）
  - 成果物の保存先について指示がある場合は、指定された箇所に書き込む
  - 指示がない場合は、通常のレスポンスとして返答の後に続けて出力する
- **作業**: システムへの変更や実行など、副作用を伴う作業

本コマンドの出力：

**返答**: 観点の抽出完了と確認依頼を報告する

**成果物**: `## 分析の観点` セクションの本文をレスポンスとして出力する

**作業**: なし

## 目的

- タスク固有の特別な観点・価値観を抽出する
- 解決方針を検討する際の評価軸を明確化する
- 一般的な観点は基本的に除外し、このタスクならではの観点に焦点を当てる

## 入力で確認すること

- 対象タスクのリンクと概要
- タスクの背景情報（task-backgroundで整理された内容）
- 前提条件・制約（task-conditionsで整理された内容）
- 組織やプロジェクトの方針・価値観
- 情報が不足している場合は、着手前に質問を箇条書きする

## 手順

### 1. 背景・制約情報の確認

- task-backgroundで整理された背景・目的を確認
- task-conditionsで整理された制約を確認
- 組織やプロジェクト固有の方針・価値観を読み取る

### 2. 観点の抽出

**確実な観点**（背景・制約に明示的に書かれているもの）を抽出：
- タスク固有の観点のみを抽出
- 一般的な観点（セキュリティ、コスト、パフォーマンス、保守性など）は基本的に除外
- ただし、一般的な観点でもこのタスク固有の特別な理由で重要な場合は含める
  - 例：「金融系のため、特にコンプライアンス対応が必須」
  - 例：「リアルタイム性が最重要のため、レイテンシーを最優先」

**推測した観点**（暗黙的だが重要そうなもの）を推測：
- 背景情報から推測できるタスク固有の観点
- 組織の文化や過去の意思決定から推測される価値観

### 3. ユーザーへの確認

AskUserQuestionツールで以下を確認：
- 推測した観点は正しいか
- 他に重要な観点はないか
- 必要に応じて優先度を確認

### 4. 観点の整理と確定

- ユーザーの回答を反映
- 最終的な観点リストを整理
- 各観点について「なぜこのタスクで重要か」を一文で簡潔に説明

## 成果物例

```
## 分析の観点

- **ベンダーロックインの回避**
  - AIのモデルを特定ベンダーに依存させたくない

- **既存ツールとの統合性**
  - チケット管理システムやリポジトリとシームレスに連携できること

- **新規ツール導入コストの最小化**
  - 既存の運用フローを大きく変えず、新しいSaaSの学習コストを抑えたい

- **データ主権の確保**
  - AIの学習データや実行ログを自組織の管理下に置きたい
```
